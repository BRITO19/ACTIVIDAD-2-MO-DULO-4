ACTIVIDAD 2 MÓDULO 4 (Análisis y diseño de sistemas)

  1. Elabora un conjunto de normas y estándares que deben seguirse en el desarrollo de software para garantizar la consistencia en la codificación y la documentación. Proporciona ejemplos concretos de cómo se aplicarían estas normas en la práctica.

    2. 1. Normas de Codificación

    1.1. Estilo de Codificación

    • Convenciones de Nombres:
    • Variables: Usar camelCase (ej. totalAmount).
    • Funciones y Métodos: Usar camelCase (ej. calculateTotal()).
    • Clases y Tipos: Usar PascalCase (ej. OrderProcessor).
    • Constantes: Usar UPPER_SNAKE_CASE (ej. MAX_RETRY_COUNT)

    Formato del Código:

    • Indentación: Usar 2 o 4 espacios (nunca tabulaciones). Definir en el archivo .editorconfig.
    • Líneas: Mantener las líneas de código a un máximo de 80-120 caracteres.
    • Espaciado: Usar espacios alrededor de operadores y después de comas, pero no antes de puntos y comas.

    1.2. Estructura del Código

    • Longitud de Métodos: Los métodos deben ser cortos y realizar una sola tarea. Idealmente, no más de 20-30 líneas.
    • División de Clases: Una clase debe tener una única responsabilidad. Usar la técnica de Single Responsibility Principle (SRP).
    • Comentarios: Comentar el código que no es obvio. Usar comentarios TODO para tareas pendientes y FIXME para errores conocidos.

    1.3. Manejo de Errores

    • Excepciones: Manejar las excepciones de manera específica y no genérica. No capturar Exception en bloques amplios.
    • Registro de Errores: Implementar un sistema de logging para registrar errores y eventos importantes.

    1.4. Control de Versiones

    • Mensajes de Commit: Los mensajes deben ser claros y descriptivos. Usar el formato de "Tipo: Descripción" (ej. Fix: Corregir error en la autenticación).
    • Branches: Usar ramas para cada característica o corrección de errores, y seguir la convención feature/, bugfix/, hotfix/.

    2. Normas de Documentación

    2.1. Documentación del Código

      Comentarios Inline:

    • Funciones y Métodos: Describir qué hace la función, los parámetros y el valor de retorno
    • Clases: Describir la responsabilidad de la clase y los métodos principales

      Documentación API:

    • Formato: Usar comentarios en formato estándar como Javadoc, Doxygen, o Swagger para APIs REST.
    • Detalles: Incluir información sobre endpoints, parámetros, tipos de datos, y respuestas esperadas.

    2.2. Documentación Técnica

      Documentación de Arquitectura:

    • Descripción: Incluir diagramas y descripciones de la arquitectura del sistema, componentes, y sus interacciones.
    • Formato: Usar herramientas como UML para diagramas.

      Documentación de Diseño:

    • Descripción: Documentar el diseño de la base de datos, esquemas de datos, y flujos de trabajo.
    • Formato: Usar diagramas de entidad-relación (ERD), diagramas de flujo, y descripciones detalladas.

      Documentación de Instalación y Configuración:

    • Descripción: Incluir guías paso a paso para la instalación del sistema, configuración de entornos, y despliegue.
    • Formato: Incluir ejemplos de archivos de configuración y comandos necesarios.

    2.3. Documentación de Usuario

      Guías de Usuario:

    • Descripción: Crear guías detalladas sobre cómo usar las características del software.
    • Formato: Usar capturas de pantalla, ejemplos y tutoriales paso a paso.

    Documentación de Ayuda:

    • Descripción: Proveer una sección de ayuda dentro del software para que los usuarios puedan resolver problemas comunes.
    • Formato: Usar una combinación de texto, imágenes, y vídeos si es posible.

    3. Normas de Calidad

    3.1. Pruebas

      Cobertura de Pruebas:

    • Unidad: Todas las unidades de código deben tener pruebas asociadas.
    • Integración: Probar la integración entre componentes y sistemas externos.
    • E2E (End-to-End): Validar el flujo completo de la aplicación desde el punto de vista del usuario.
    • Herramientas: Usar herramientas de CI/CD para ejecutar pruebas automáticamente en cada commit.

    3.2. Revisión de Código

      Proceso de Revisión:

    • Peer Reviews: Realizar revisiones de código por parte de compañeros antes de fusionar cambios.
    • Checklist: Usar una lista de verificación para asegurar que se cumplan las normas de codificación y estándares de documentación.

    4. Normas de Seguridad

    4.1. Gestión de Credenciales

    • Almacenamiento: No almacenar credenciales en el código fuente. Usar servicios de gestión de secretos.
    • Acceso: Limitar el acceso a credenciales y datos sensibles a los miembros del equipo que lo necesiten.

    4.2. Validación de Entrada

    • Sanitización: Sanitizar todas las entradas del usuario para prevenir ataques de inyección.
    • Validación: Validar las entradas del usuario en el servidor y el cliente.

    4.3. Seguridad en el Código

    • Dependencias: Usar herramientas de análisis de seguridad para identificar vulnerabilidades en las dependencias de terceros.
    • Actualización: Mantener las bibliotecas y dependencias actualizadas con los últimos parches de seguridad.
